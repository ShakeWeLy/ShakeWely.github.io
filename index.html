1 ReAct 模式

https://github.com/samwit/langchain-tutorials/blob/main/agents/YT_Exploring_ReAct_on_Langchain.ipynb

原理

  1. 接收任务：Agent获取来自用户或系统的任务指令。
  2. 推理（Thought）：Agent依据当前任务和已有知识展开推理，形成初步的行动计划。
  3. 行动（Action）：Agent执行经推理得出的行动。
  4. 观察（Observation）：Agent对行动产生的结果进行观察，获取反馈信息。
  5. 循环迭代：将观察结果反馈至推理过程，Agent依据新信息重新推理，生成新的行动计划，并持续执行行动与观察，直至任务完成

2 Plan and solve 模式

《Plan-and-Solve Prompting: Improving Zero-Shot Chain-of-Thought Reasoning by Large Language Models》

https://github.com/Fosowl/agenticSeek

原理

  1. 接收任务：Agent接收来自用户或系统下达的任务指令。

  2. 规划（Plan）：Agent基于任务目标生成一个多步骤的详细计划，清晰界定每个步骤的具体内容与执行顺序。

  3. 执行（Solve）：Agent依照既定计划，依次执行每个步骤。

  4. 观察（Observation）：Agent对执行结果展开观察，判断是否有必要启动重新规划流程。

  5. 重新规划（Replan）：若发现计划无法实施，或需要根据实际情况调整，Agent将结合当前状态重新生成计划，并继续执行。

3. Reason without Observation

  1. Planner：负责生成一个相互依赖的“链式计划”，定义每一步所依赖的上一步的输出。
  2. Worker：循环遍历每个任务，并将任务输出分配给相应的变量。当调用后续调用时，它还会用变量的结果替换变量。
  3. Solver：求解器将所有这些输出整合为最终答案。

4. LLMCompiler

原理

变成并行执行

  1. 接收任务：Agent 接收到用户或系统的任务指令，任务可能包含多个子任务。

  2. 任务分解（Task Decomposition）：Agent 将复杂任务分解为多个子任务，并确定这些子任务之间的依赖关系。

  3. 并行执行（Parallel Execution）：Agent 根据子任务之间的依赖关系，将可以并行处理的子任务同时发送给多个执行器进行处理。

  4. 结果合并（Result Merging）：各个执行器完成子任务后，Agent 将结果合并，形成最终的输出。

  5. 循环迭代（Iteration）：如果任务需要进一步处理或调整，Agent 会根据当前结果重新分解任务，并继续并行执行和结果合并，直到任务完成。

computed 是什么？

computed 是 Vue 中的计算属性，它的作用是根据其他数据自动计算出一个新的值。

简单来说，computed 就像一个自动计算的公式，当依赖的数据变化时，它会自动重新计算。

指令微调的语言模型

普通语言模型只学会了“在大语料里续写文本”；
指令微调模型进一步学会了：

“当我说 请翻译成中文 时，不是续写‘请翻译成中文’，而是执行翻译任务。”

基于人类反馈的强化学习（RLHF，Reinforcement Learning with Human Feedback）

https://chatgpt.com/s/t_6870b6c17a5081919a2de618af9c434b

RLHF 是一种让语言模型通过人类偏好来“对齐”行为的技术：
它不是仅仅预测下一个词，而是学会“什么样的回答更符合人类预期”。

预训练语言模型（如 GPT）是通过大量文本数据学习的，它只知道“语言模式”，不知道什么是“好回答”、“有害内容”或“礼貌回应”。

🔧 RLHF 的目标是：
在模型已经具备生成能力的基础上，用人类反馈进一步调整模型行为，让它：

  • 更有用（helpful）
  • 更无害（harmless）
  • 更真实（honest）
  • 更对齐（aligned）

✅ RLHF 三阶段工作流程（标准流程）

这就是 OpenAI 的 InstructGPT / ChatGPT 的训练流程，分为三个阶段：

🔵 Step 1：监督微调（Supervised Fine-Tuning, SFT）

  • 用“指令+人类优质响应”对模型做初步微调。
  • 例如：

✅ 模型从“语言模型”变成“跟人类说话的模型”。

🔵 Step 2：奖励模型（Reward Model, RM）训练

  • 收集人类对模型多个回答的偏好排序数据：

    输入一个问题，模型生成多个答案 → 人类打分 → 哪个更好？

  • 训练一个 Reward Model（通常是 BERT/Bloom 类模型），学会判断“哪个回答更好”。

✅ 模型能评估回答质量（越像人，reward 分越高）。

🔵 Step 3：强化学习（PPO,DPO等）优化原始模型

  • 使用策略优化算法（如 PPO：Proximal Policy Optimization）
  • 目标是让模型生成更高 Reward 的答案
  • 奖励来自 Reward Model，而不是传统“环境”

✅ 模型行为被“人类价值”引导，而不是语言概率最大。

对齐(Alignmen)

技术 简介
DPO（Direct Preference Optimization） 替代 RLHF，直接用人类偏好做梯度优化，更高效
Constitutional AI（宪法式对齐） Anthropic 提出，模型通过预设“宪法”规则自我审查和自我学习，无需人工打分
Self-Reward / Self-Alignment 让模型基于提示自评行为（如 OpenAI 的 Critic 模块）
多轮互动对齐（Multi-turn alignment） 加强模型在长对话、多轮引导下保持对齐状态
Instruction Backchaining 自动生成训练用的指令和行为对，用于自监督对齐

[!info]

非常好，这里我给你总结一个通用的“对比试验”流程，不只适用于这次的 字幕翻译对比，也适用于你以后做任何 A/B 测试、模型对比、工具对比时使用。

✅ 对比试验 — 一般流程

1️⃣ 明确目的

  • 你要比较什么？
    • 👉 两个（或多个）方案/工具/模型
  • 想验证什么？
    • 👉 哪个更准确、更快、更省事、更稳定

2️⃣ 确定对比指标

  • 常用指标：
    • 准确性
    • 自然度
    • 速度
    • 易用性
    • 输出格式
    • 资源占用（CPU/GPU/RAM）
  • 用表格列出来

3️⃣ 准备统一的输入

  • 样本要可控、代表性强
  • 文件格式一致
  • 大小覆盖典型场景（短/长/简单/复杂）
  • 数量合适（一般 3~5 个就够）

4️⃣ 执行试验

  • 用同样的输入，分别跑每个方案
    • 保证参数一致（同一个模型/同样 chunk）
    • 记录运行环境（GPU/CPU/网速）
  • 保存原始输出

5️⃣ 收集结果

  • 每个样本、每个方案一一对应
  • 命名清晰（A_short_Buzz.srt / A_short_Hgface.srt）
  • 保留日志（耗时、错误）

6️⃣ 对比分析

  • 使用对比工具（Diffchecker、Notepad++、Excel）
  • 核查：
    • 相同之处（共性）
    • 不同之处（误差、漏译）
  • 人工或自动统计准确率

7️⃣ 记录并评分

  • 用表格记录每个指标打分（1~5）
  • 标记特别差或特别好的点
  • 可以加截图举例

8️⃣ 得出结论

  • 汇总：
    • 哪个方案总体更好
    • 哪个场景适合哪个方案
  • 写一段总结

9️⃣ 可选：做复验证

  • 用别的文件再跑一轮
  • 检查结果是否一致
  • 如果一致，结论更可信

10️⃣ 归档

  • 保存：
    • 输入文件
    • 输出文件
    • 对比表
    • 报告文档
  • 方便以后复查

📌 总结：一图流

🎯 专注要点

✔ 输入一致
✔ 参数一致
✔ 对比维度明确
✔ 可重复、可复现
✔ 有记录、有报告

🗂️ 如果需要，我可以帮你做：

✅ 通用对比表格（Excel、Notion、Markdown 版）
✅ 可复用试验报告模板
✅ 自动批处理脚本（一次跑完多文件）

Step 1️⃣ 选定实验文件

至少准备：

  • 短文件 1~2 个（30 秒 ~ 1 分钟）
  • 长文件 12 个（510 分钟）
  • 尽量有口音、重复词、停顿，增加难度

Step 4️⃣ 对比重点

维度 具体看什么
翻译准确性 有无漏译、误译
语义自然度 是否直译生硬
专有名词 是否翻译一致
句式流畅 是否有不通顺
长句子处理 是否断句合理
分段完整性 Buzz 常自动分段，Hgface 默认整段

Step 6️⃣ 做表格记录

文件 长短 Hgface 翻译评分 (1~5) Buzz 翻译评分 (1~5) 差异点
A 短 4 5 Buzz 句式更口语
B 长 3 4 Hgface 连贯度差
C 短+口音 2 3 都识别漏词
D 长+专业 3 4 Buzz 漏翻术语
