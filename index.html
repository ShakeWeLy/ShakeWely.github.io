2025年12月24日 agent记忆管理

Agent记忆管理

  • 短期记忆：类似于我们临时记住一个电话号码，过一会儿就可能忘记。大模型通过在输入中嵌入提示语句（prompt），借助自身强大的上下文理解能力来“记住”最近的对话内容。但这种记忆会随着上下文窗口的限制而被覆盖或遗忘。
  • 长期记忆：更像是我们记住一段旅行经历，或者对某个朋友的长期印象。智能体通过建立外部记忆库来实现知识和经验的持久化存储，可以在需要时随时检索出来，帮助它更高效地解决问题

0 本质

1. 核心机制（你已点明的部分）

  • 序列化：将对话历史、知识、用户画像等压缩进有限上下文（Prompt）。
  • 检索：从向量库、数据库或记忆中查找相关信息。
  • TopK：选择最相关的几条信息。
  • 拼接：将检索结果与原问题拼接，送入模型。

2. 超越“简单拼接”的关键演进

虽然基础操作类似，但先进系统在以下方面做了大量工作：

a. 记忆的层次与结构

  • 短期记忆：当前会话的上下文（直接拼接）。
  • 长期记忆：
    • 向量记忆：语义搜索，处理非结构化知识。
    • 图记忆：存储实体、关系，用于推理（如知识图谱）。
    • 结构化记忆：SQL/键值存储，存储精确信息（如用户偏好）。
    • 程序性记忆：让LLM学会调用工具、API，扩展能力边界。

Memory Bank

MemoryBank 包含三个核心模块：

  1. 记忆存储：保存对话记录、事件总结和用户个性分析，就像建立一个多层次的记忆档案。
  2. 记忆检索：根据上下文快速找到相关的“回忆”。它使用一种稠密向量检索方法（类似 Dense Passage Retrieval, DPR），把每条记忆片段编码成向量，再用 FAISS 进行高效搜索。

2025年12月22日 ConvNeXt

ConvNeXt

从传统 ResNet 逐渐演变到具有Transformer特点的卷积神经网络的演变路径，仍然基于卷积神经网络架构，但它在设计上融入了一些Transformer的元素或特性，使得它在某种程度上与Transformer相似，但仍然保持了ConvNet的基本性质。

https://zhuanlan.zhihu.com/p/1928810982915941594

1 超参数调优：重塑 ResNet

ConvNeXt 的核心思想其实就是：对 ResNet 架构进行系统性的超参数调优。具体来说，作者主要从五个方面进行了实验探索：

  1. 宏观结构设计（Macro Design）
  2. 引入 ResNeXt 模块（ResNeXt）
  3. 反向瓶颈结构（Inverted Bottleneck）
  4. 使用大卷积核（Large Kernel）
  5. 微观结构设计（Micro Design）

宏观设计（Macro Design）

将 ResNet 的阶段比例（stage ratio）调整为与 Swin-T 架构一致，也就是 1:1:3:1。

ResNeXt 化（ResNeXt-ification）

主要是在 ResNet 的 bottleneck 模块中引入了组卷积（group convolution）

倒置瓶颈结构（Inverted Bottleneck）

ResNet属于 “宽 → 窄 → 宽” 的布局，因此被称为“瓶颈结构”。但 ConvNeXt 并没有采用这个经典设计，而是引入了“倒置瓶颈”（Inverted Bottleneck），也就是 “窄 → 宽 → 窄”，这个结构灵感来自 Transformer 架构中的前馈网络层。

卷积核大小

让深度卷积（depthwise convolution）放到了模块的开头，这一设计灵感来源于 Transformer 结构，在 Transformer 中多头自注意力层（MSA）会放在前面，后面才是前馈网络层（MLP）。

微观设计

论文中调优的最后一个方面是“微观设计”，主要是网络细节层面的调整。和之前一样，这些参数大多借鉴了 Transformer 的设计。首先，作者用 GELU 激活函数替代了 ReLU。虽然替换后准确率保持不变（仍是 80.6%），但他们决定在后续实验中采用 GELU。

之后，他们发现减少激活函数的使用次数能提升准确率。具体做法是，不再在 ConvNeXt 块中的每个卷积层后都使用 GELU，而只在两个点卷积层之间使用一次。这样一改，准确率提升到了 81.3%，已经追平了 Swin-T 架构，而且计算复杂度更低（4.2 vs 4.5 GFLOPS）。

接下来是批量归一化（Batch Normalization，BN）层的调整。传统 CNN 架构常用 Conv-BN-ReLU 结构，ResNet 也沿用了这个设计。但作者们选择只用一层 BN，放在第一个点卷积层之前。这个改动使准确率微升至 81.4%，略微超过了 Swin-T。

尽管如此，参数调优还没结束，接着用层归一化（Layer Normalization）替代了 BN，准确率再次提升了 0.1%，达到 81.5%。

所有这些微观设计的改动最终汇总成了图 5（最右侧）的架构示意图。从图中可以清楚看出 ConvNeXt 块与 Swin Transformer 及 ResNet 块的不同之处。

方式 A：copy环境: Electron 启动独立 Python 进程

1. venv 的原理

venv 并不是完全自包含的 Python，它依赖创建时 系统 Python 的文件来构建：

  • Scripts/python.exe → 指向 venv 内的可执行文件（拷贝或软链接）
  • Lib/ → 包含标准库
  • pyvenv.cfg → 记录 home（原系统 Python 位置）和 executable

关键是：

当你在其他机器运行 venv：

  • 只要 venv 内的 python.exe 和 Lib 完整，路径不同一般不会报错。
  • 如果 venv 内缺少标准库或 DLL，就会报错 ModuleNotFoundError: encodings 或启动失败。

vеnv 依赖系统 Python 的路径（pyvenv.cfg 指向打包路径）

  • home 指向你本地的系统 打包的Python路径
  • 客户机上路径不同 → venv 找不到标准库 → 报错 ModuleNotFoundError: encodings
  • ⚠️ 这是之前遇到的错误的根本原因 → 安装新环境并修改对应的path文件

这样运行会依赖创建venv的python

本质上依赖于系统 Python，因为 venv 只是拷贝/软链接系统 Python 的部分文件（可执行文件 + 标准库）。所以：

  1. 如果用户机器没有 Python，或者版本不对，Electron 打包后的程序 无法独立运行。

根据VLM的输入处理和输出生成能力将其分为三个不同的组：

  • 视觉语言理解模型：专门为视觉信息与语言的解释和理解而设计的模型. 比如CLIP

1.LLaVA

详解视觉语言模型原理及代码，以DeepSeek-VL为例

通过建立统一的多模态表示空间，将不同模态的信息转换为统一的token表示，然后在同一个架构中进行联合建模和推理。既然大语言模型擅长处理token序列，那么就把所有模态的信息都转换成token。

视觉token：将图像切分为固定大小的patches（如16×16像素块），每个patch编码为一个视觉token
语言token：保持传统的subword tokenization，每个词汇对应一个语言token
统一建模：将视觉token和语言token拼接成统一序列，用同一个Transformer架构处理

视觉编码编码器

视觉编码器用于将图像转换为向量表示，在DeepSeek-VL中，图像被视觉编码器转换为576个向量（图像的token embedding）。VLM的视觉编码器直接使其他模型预训练好的参数，普遍使用的视觉编码器结构为ViT（Vision Transformer），但可能是不同方式训练出来的，例如DeepSeek-VL使用的是Siglip和SAM训练出来的ViT，而Qwen-VL使用的是OpenCLIP的ViT。

视觉适配器

视觉适配器用于将用于将ViT的输出图像的token embeeding映射到与文本embedding相同的空间，便于让LLM理解图像中的内容。常见的结构有多层MLP、cross attention等。

LLM

LLM是VLM的核心，视觉编码器和适配器最终产出的图像的token embedding都是要输入到LLM进行理解的，并由LLM输出关于图像的回答。
