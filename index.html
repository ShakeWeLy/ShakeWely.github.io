python环境打包构建

在 Electron + Python 项目中，为了在无 Python 环境的客户端运行，需要实现 Python 环境自包含打包。
实践过程中共尝试三种方案：

  1. 方式 A：复制本地 Python + venv
  2. 方式 B：PyInstaller 打包成独立 EXE
  3. 方式 C：Embeddable Python（嵌入式解释器）

最终发现仅方案 B 可行，A、C 在兼容性和安装层面存在根本性限制。

方式 A：copy环境: Electron 启动独立 Python 进程

1. venv 的原理

venv 并不是完全自包含的 Python，它依赖创建时 系统 Python 的文件来构建：

  • Scripts/python.exe → 指向 venv 内的可执行文件（拷贝或软链接）
  • Lib/ → 包含标准库
  • pyvenv.cfg → 记录 home（原系统 Python 位置）和 executable

关键是：

当你在其他机器运行 venv：

  • 只要 venv 内的 python.exe 和 Lib 完整，路径不同一般不会报错。
  • 如果 venv 内缺少标准库或 DLL，就会报错 ModuleNotFoundError: encodings 或启动失败。

vеnv 依赖系统 Python 的路径（pyvenv.cfg 指向打包路径）

  • home 指向你本地的系统 打包的Python路径
  • 客户机上路径不同 → venv 找不到标准库 → 报错 ModuleNotFoundError: encodings
  • ⚠️ 这是之前遇到的错误的根本原因 → 安装新环境并修改对应的path文件

2.问题分析:

原因: cursor 给我错的步骤, 以为做了形象工程: copy和pip了新的环境, 但是运行的确实本地的环境的python, 导致后续一直在错的分支上修改(不了解本质, 一直以为是python copy的错误, 见下面分析).

2.1 自包含方式python错误

这样运行会依赖创建venv的python

本质上依赖于系统 Python，因为 venv 只是拷贝/软链接系统 Python 的部分文件（可执行文件 + 标准库）。所以：

  1. 如果用户机器没有 Python，或者版本不对，Electron 打包后的程序 无法独立运行。
  2. 同时, 打包的pyvenv.cfg 指向了本地机的python导致客户机的python找不到,

**核心问题 : **

  • venv 本身不是自包含的，它只是一个「隔离的环境」，依赖底层 Python。
  • 要想 Electron 完全自包含，你必须用 Embedded Python 或 完整 Python 安装 + Lib，而不是 venv。之前手动 copy python.exe + DLL + venv，这本质就是想绕开系统 Python，但做法容易出错（缺标准库，导致 encodings 不存在）

方式 C：Embeddable Python（嵌入式） ❌

❌❌很多库都安装不了

Cursor 分步安装的原因 跳过了包的安装导致缺少库, 运行不了

1. pip 安装问题

embeddable python不带pip需要手动下载和修改

需要删除import site在python312._pth文件中

2. embeddable python特性:

2.1 库编译问题

  1. 依赖
    • opencv-python 本身提供了预编译 wheel，但有些依赖（如 matplotlib）可能需要 源代码编译，尤其在缺少 wheel 或版本不匹配的情况下
    • matplotlib 依赖 cycler、kiwisolver、pillow 等，某些组件可能需要 C/C++ 编译工具
    • mesonpy 是最近 matplotlib 构建系统的一部分，用于编译扩展
  2. 嵌入式 Python 不包含编译环境
    • 没有 Visual Studio C++ Build Tools
    • 没有 python-config、make、ninja 等工具

所以在 嵌入式 Python 下 pip install opencv-python 或 matplotlib 就会失败，因为它需要编译环境，嵌入式 Python 只能提供一个纯净的解释器和标准库。

2.2可行解决方案

  1. 使用预编译 wheel（

    • 下载适合你 Python 版本和架构的 .whl 文件，直接安装：

    • 这样就不用本地编译了。

    • 你可以在 PyPI opencv-python 找到对应 wheel。

  2. 下载相关编译工具集

3. 问题分析：

3.1 matplotlib 编译失败

  1. matplotlib 编译失败：matplotlib==3.10.1 需要从源码编译，但缺少 mesonpy 构建工具

  2. mesonpy 后端不可用：Cannot import ‘mesonpy’ - 构建工具链不完整

  3. 依赖解析失败：pip 无法找到兼容的包版本

解决方案：

  1. 特殊处理 matplotlib：

  • 优先使用预编译的 wheel 包：–only-binary=:all: --prefer-binary

  • 如果指定版本失败，尝试不指定版本的最新版本

3.2 工具链安装失败

3.3 torch 安装失败

这是因为嵌入式Python环境可能缺少某些系统库，导致torch无法安装。

  • 需要手动编译!!

3.2. 依赖matplotlib的opencv, ultralytics安装失败

  1. 下载相关编译工具
  2. ultralytics 8.0.2 depends on matplotlib>=3.2.2

3. ultralytics版本高要求新版本的matplotlib

内部库依赖版本冲突无交集

  1. 不指定版本
  2. 使用更宽松的安装策略
  3. torch要求

3.av库版本问题(webrtc)

无法找到合适版本

  1. 修改版本
  2. 不指定版本
  3. 暂时不安装了

4.总结

嵌入式 Python 自带python, 但是很多包和库基本上pipy都不支持, 导致无法安装完整的环境, 这反而失去了python的优势, 所以放弃改方案

打包总结

1. venv 并非独立环境

  • venv 本质上是系统 Python 的隔离副本；
  • pyvenv.cfg 中的 home 和 executable 会绑定本地 Python；
  • 换机后路径失效导致运行错误；
  • 结论：venv 不适合打包分发。

1.LLaVA

详解视觉语言模型原理及代码，以DeepSeek-VL为例

通过建立统一的多模态表示空间，将不同模态的信息转换为统一的token表示，然后在同一个架构中进行联合建模和推理。既然大语言模型擅长处理token序列，那么就把所有模态的信息都转换成token。

视觉token：将图像切分为固定大小的patches（如16×16像素块），每个patch编码为一个视觉token
语言token：保持传统的subword tokenization，每个词汇对应一个语言token
统一建模：将视觉token和语言token拼接成统一序列，用同一个Transformer架构处理

视觉编码编码器

视觉编码器用于将图像转换为向量表示，在DeepSeek-VL中，图像被视觉编码器转换为576个向量（图像的token embedding）。VLM的视觉编码器直接使其他模型预训练好的参数，普遍使用的视觉编码器结构为ViT（Vision Transformer），但可能是不同方式训练出来的，例如DeepSeek-VL使用的是Siglip和SAM训练出来的ViT，而Qwen-VL使用的是OpenCLIP的ViT。

视觉适配器

视觉适配器用于将用于将ViT的输出图像的token embeeding映射到与文本embedding相同的空间，便于让LLM理解图像中的内容。常见的结构有多层MLP、cross attention等。

LLM

LLM是VLM的核心，视觉编码器和适配器最终产出的图像的token embedding都是要输入到LLM进行理解的，并由LLM输出关于图像的回答。
